1.1 Mission Statement
The Angela Coaching Engine is an adaptive decision framework designed to simulate elite human coaching judgment using structured physiological models.
It interprets endurance athlete data (load, performance, recovery) through deterministic rules to yield prescriptive and interpretive outputs.
The purpose: maintain adaptive progression with controlled fatigue across multi-sport training blocks leading to peak performance at target events.

1.2 Core Principle
“Load must be absorbed before it can be expressed.”
All logic in the engine enforces this: load creates adaptation only when recovery, readiness, and stress balance align within defined parameters (CTL, ATL, TSB).

1.3 Core Functions

2.2 Derived Metrics (Equations)
2.2.1 Exponential Weighted Moving Averages
CTLₜ = CTLₜ₋₁ + (TSSₜ – CTLₜ₋₁) * (1 / 42)
ATLₜ = ATLₜ₋₁ + (TSSₜ – ATLₜ₋₁) * (1 / 7)

These represent 42-day and 7-day response constants, equivalent to training impulse models (Bannister).
2.2.2 Training Stress Balance
TSBₜ = CTLₜ₋₁ – ATLₜ₋₁

Positive TSB → Freshness
Negative TSB → Fatigue
Rapid change (ΔTSB > 20 in 3 days) → instability flag
2.2.3 Load Momentum (ΔCTL)
LoadMomentum = (CTLₜ – CTLₜ₋₇) / 7

Indicates rate of fitness change (points per day).
• > +1.0 → strong progression • < −0.5 → detraining
2.2.4 Fatigue Index
FatigueIndex = (ATL / CTL) – 1

• > 0.10 → over-reaching
• < −0.10 → under-stimulated
2.2.5 Recovery Coefficient
RecoveryCoef = clamp( (TSB + 25) / 50 , 0 , 1 )

Scaled 0–1 to weight recovery and fueling advice.
2.2.6 Readiness Score (Composite 0-100)
Readiness = 100 –
 ( |FatigueIndex| * 50 ) –
 (max(0, -TSB) * 0.8 ) –
 (SleepPenalty + MoodPenalty + HRVPenalty)

Where
• SleepPenalty = (5 – SleepQuality) × 2
• MoodPenalty = (max(0,1–MoodIndex) × 5)
• HRVPenalty = (if HRV < baseline × 0.9 then 10 else 0)
Readiness < 60 → limit intensity.
Readiness > 85 → eligible for key session or test.
2.2.7 Durability Score
Durability = (Z1+Z2 TSS share) / TotalTSS

Fraction of aerobic workload. Durability > 0.70 → strong aerobic foundation.

PART III – STRESSLOGIC MODEL

3.1 Purpose
The StressLogic model is the interpretive core of Angela’s brain.
It transforms the raw metrics layer (CTL, ATL, TSB, HRV, subjectives, travel) into qualitative states that a coach would describe in natural language:
“You’re carrying productive fatigue,”
“You’re in deep overreach,”
“You’re ready for intensity,”
“You need two easy days before testing.”
Every decision downstream — block choice, session type, fueling and recovery instructions — originates here.

3.2 Fundamental Principle
Stress ≠ Fatigue.
Stress → fatigue → adaptation → fitness → form.
StressLogic seeks to maintain the athlete between productive stress and accumulative fatigue.
This zone corresponds to roughly TSB = –10 to –25 and FatigueIndex = 0 to 0.15.

3.3 Inputs
All inputs are normalised 0–1 internally before rule evaluation.

3.4 Primary States
Each state persists until metrics exit the defined thresholds for 48 h.

3.5 Logic Flow
Pseudocode representation of the decision stack:
if TravelFlag = True:
    apply TravelPenalty
if PostRace < 14d:
    state = Rebuild
else:
    if TSB < -40 or HRV < 0.8*baseline:
        state = Compromised
    elif TSB < -25 or FatigueIndex > 0.15:
        state = Overreached
    elif -25 <= TSB <= -10:
        state = ProductiveFatigue
    elif TSB > +10 and CTL increasing:
        state = Recovered
    elif TSB > +25 and CTL decreasing:
        state = Detraining

Secondary modifiers then adjust confidence using subjectives.

3.6 Secondary Modifiers
Sleep Modifier
 If SleepQuality ≤ 3 → decrease readiness × 0.9
Mood Modifier
 If Mood ≤ –1 → readiness × 0.85
Motivation Modifier
 If Motivation ≥ +1 and TSB < –10 → shift state upward one level (“productive” → “ready”)
Jet-Lag Adjustment
 If TravelFlag = True and timeZones > 5 → reduce all intensity coefficients × 0.5 for 72 h

3.7 Output Variables

3.8 Example Interpretation Rules

3.9 Integration With Training Blocks
Productive Fatigue → maintain current block (Base, Build, VO₂, Specificity).
Overreached → auto-switch to Rebuild (1–2 wks).
Compromised → suspend all intensity, flag illness).
Recovered → eligible for progression / testing).
Detraining → trigger Base or Build reset).

3.10 Adaptive Load Response
StressLogic continuously recalibrates future TSS targets:
if state == ProductiveFatigue:
    TargetTSS = PrevWeekTSS × 1.00
elif state == Recovered:
    TargetTSS = PrevWeekTSS × 1.05
elif state == Overreached:
    TargetTSS = PrevWeekTSS × 0.85
elif state == Compromised:
    TargetTSS = PrevWeekTSS × 0.6

This feeds directly into the TSS Planner (Part IV).

3.11 Contextual Overrides

3.12 Cross-Link With Fueling Engine
When StressState ∈ {Overreached, Compromised}, fueling rules double the post-session CHO recommendation (× 1.5) and flag “restore glycogen first”.

3.13 Narrative Mapping
Each state maps to tone and wording templates used when Angela writes summaries:

3.14 Summary
StressLogic is the translator between metrics and human-readable coaching language.
It maintains dynamic equilibrium between stress and recovery.
It sets flags for block transitions and modulates fueling and intensity.
It is deterministic, auditable, and can run daily without manual input.

PART V – TRAINING BLOCK LOGIC AND STRUCTURE

5.1 Purpose
Blocks serve as discrete training phases, each targeting a specific physiological limiter (aerobic base, threshold power, aerobic ceiling, execution, recovery, or hybrid).
They ensure directed adaptation and avoid chronic stagnation by rotating emphasis every 6–10 weeks.
Each block defines:
Duration range
Intensity and volume rules
Target fatigue profile
Key workouts
Exit criteria

5.2 Block Overview Table

5.3 Block Entry Logic
Block selection comes directly from the Decision Framework (Part VI), but entry conditions can also trigger from the StressLogic layer:

5.4 Base / Durability Block
Duration: 6–10 weeks
Goal: Improve durability, consistency, and aerobic resilience.
Zone Distribution:
Key Workouts:
Bike: 3×12 @ 85–88% CP
Run: 3×8 min uphill tempo @ 95–100% CS
Long Ride: 2.5–4h @ Z2
Long Run: 75–100 min Z1–Z2
Exit Criteria:
CTL stabilized (> 6 wk upward trend)
No tissue fatigue
HRV baseline restored
Easy days feel easy

5.5 Build / Threshold Block
Duration: 6–10 weeks
Goal: Raise CP/CS; improve sustained power/pace and fatigue resistance.
Zone Distribution:
Key Workouts:
Bike: 3×12–15 @ 92–95% CP
Run: 2×15 @ 98–102% CS
Long Bike: 3–4h w/ 2×20 @ 88% CP
Brick: 2.5h bike + 30m run @ 90% CP/CS
Exit Criteria:
CP/CS increased ≥ 3–5%
Subjective “snap” present
HRV baseline recovered

5.6 VO₂ Max Block
Duration: 4–8 weeks
Goal: Lift aerobic ceiling, boost repeatability at >110% CP/CS.
Zone Distribution:
Key Workouts:
Bike: 5×4 @ 110–112% CP
Run: 6×3 @ 110–115% CS
Swim: 12×100 @ CSS+5”
Exit Criteria:
VO₂ tolerance improved
CP:VO₂ ratio < 0.85 → → >0.88
Fatigue index >0.15 → resolved

5.7 Specificity Block
Duration: 4–6 weeks
Goal: Translate fitness into execution at race pace; rehearse fueling & environment.
Zone Distribution:
Key Workouts:
Bike+Run Brick: 2.5h @ 80–88% CP + 30m @ 90% CS
Run: 3×10 progressive 90→100% CS
Swim: Race simulation 2000–3000m continuous
Exit Criteria:
Race pacing automatic
Fueling practiced and validated
Fatigue stable; confidence high

5.8 Hybrid Block
Duration: 6–10 weeks
Goal: Address multiple limiters; mixed focus (e.g., Base + Threshold).
Zone Distribution:
Alternates weekly between Base pattern and Build pattern.
Key Rule:
One “Primary Focus” (60% of sessions) and one “Secondary Focus” (40%).
E.g. Base+VO₂ Hybrid → 2 VO₂ sessions/wk + rest aerobic.
Exit Criteria:
Limiter corrected
Readiness stable
TSB oscillation reduced (<15 range)

5.9 Rebuild / Reset Block
Duration: 1–3 weeks
Goal: Restore readiness, hormonal balance, and neural freshness.
Zone Distribution:
Key Workouts:
45–60m easy aerobic spin or jog
Technique swim 1500–2000m
Yoga or strength mobility
Exit Criteria:
HRV > baseline
TSB ≥ 0
Sleep and motivation restored

5.10 Block Transition Rules

5.11 Testing Protocol Integration
Before new block: 3–6–12 (bike) or 400–800–3200 (run).
Mid-block: Verify adaptation (CP or CS ↑).
Post-block: Update zones, recalc TSS scaling.

5.12 Block Narrative Summaries
Angela uses tone + narrative mapping per block to maintain athlete engagement:
Base: “We’re laying the foundation — volume steady, fatigue manageable.”
Build: “We’re lifting your sustainable power — get comfortable with controlled discomfort.”
VO₂: “Short, sharp efforts now — this raises the ceiling.”
Specificity: “Now we’re rehearsing race day — no new fitness, only precision.”
Hybrid: “Balanced work, staying sharp without excess load.”
Rebuild: “Time to recharge the system — easy doesn’t mean wasted.”

PART VI – FUELING & NUTRITION SUBSYSTEM

The engine’s objective is to keep glycogen availability and recovery in sync with TSS load.

6.2 Core Concept
Fueling Window = 30 min before + during + 30 min after.
All fueling calculations are applied inside this window.
Skipping the window decreases “Fueling Compliance Score,” which feeds back into recovery logic.

6.3 Inputs

6.4 CHO Calculation Logic
Baseline formula:
CHO_total = (SessionDuration_min × 1 g/min) + 30 g (pre) + 30 g (post)

Rounded to nearest 10 g.
For each session, the system generates:
PreCHO = 0.3 × CHO_total
DuringCHO = 0.4 × CHO_total
PostCHO = 0.3 × CHO_total

Modified by Fueling Mod (from StressLogic, 0.8–1.2).

10.7 Authentication & Permissions
OAuth2 used for TP / Strava connections.
JWT-based internal tokens for coach/athlete accounts.
Role-based control:
Coach can read all linked athletes.
Athlete can only access self-data.
Admin (you) can override all.

10.9 Communication Layer

10.10 Multi-Athlete Scaling
Each athlete’s data runs as an isolated “Angela instance” within the same container cluster.
Coaches can view summaries or open detailed sessions per athlete.
EchoDevo auto-load balances across containers using AWS ECS or Fargate.

11.5 Database Schema (PostgreSQL)
Table athletes
CREATE TABLE athletes (
  id UUID PRIMARY KEY,
  name TEXT,
  coach_id UUID,
  zones JSONB,
  current_block TEXT,
  ctl NUMERIC,
  atl NUMERIC,
  tsb NUMERIC,
  readiness NUMERIC,
  stress_state TEXT,
  fueling_score NUMERIC,
  created_at TIMESTAMP DEFAULT NOW()
);

Table workouts
CREATE TABLE workouts (
  id TEXT PRIMARY KEY,
  athlete_id UUID REFERENCES athletes(id),
  date DATE,
  sport TEXT,
  tss NUMERIC,
  duration INTEGER,
  intensity_factor NUMERIC,
  metrics JSONB,
  source TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);

Table messages
CREATE TABLE messages (
  id SERIAL PRIMARY KEY,
  athlete_id UUID,
  role TEXT,
  message TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);


11.6 AWS Deployment Steps
RDS setup – PostgreSQL 15 multi-AZ.
S3 buckets – echodevo-data (raw imports), echodevo-exports.
Secrets Manager – store all OAuth tokens.
Lambda functions –
sync_trainingpeaks (cron 6 h)
compute_daily (02:00 UTC)
compute_weekly (Sun 23:00 UTC)
ECS/Fargate – containerize core engine (Dockerfile Python 3.10 + FastAPI).
API Gateway – public entry to Lambdas and ECS.
CloudWatch – monitor execution, set alerts for error > 1 %.

11.7 Security & Compliance
All PII encrypted (AES-256 at rest, TLS in transit).
Tokens rotated every 90 days.
Role-based RBAC in Postgres (coach, athlete, admin).
GDPR / CCPA export/delete endpoints (/athlete/{id}/erase).
